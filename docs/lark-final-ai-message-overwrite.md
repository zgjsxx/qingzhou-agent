# 飞书桥接层误取最后一条 AI 消息导致最终回复被覆盖

## 问题现象

在飞书聊天中，用户提问后，Agent 实际上已经生成了完整回答，但最终回发到飞书里的内容却只剩下一句简短的收尾话术，例如：

- `总结完毕！如果还有其他想深入了解的部分……`
- `你可以在数据库中尝试用以下方式查找这张表……`

从用户视角看，表现为：

1. Agent 好像“答非所问”或“只回了半句”。
2. 真正有信息量的正文似乎丢失了。

## 根因结论

真正的回答没有丢失，而是被飞书桥接层“取最后一条 AI 消息”的逻辑覆盖掉了。

### 关键链路

1. 模型先生成一条 **有完整正文** 的 AI 消息。
2. 这条 AI 消息里同时包含了一个 `todo_write` 工具调用。
3. `todo_write` 执行完成后，模型会再进行一轮收尾。
4. 收尾这一轮通常只返回一句简短话术。
5. 飞书桥接层当前只取 `result["messages"]` 中 **最后一条非空 AIMessage**。
6. 因此最终发到飞书中的，不是第一轮的完整正文，而是最后这句简短收尾。

## 日志证据

### 案例一：RTDB 进程数据来源问题

在 `backend/logs/agent.jsonl` 中：

- `msg_ccfb8313-b389-427d-8e7a-b3a802c37333`
  - 已经给出了完整回答：
    - RTDB 进程数据来自 `modeldb`
    - 调用链 `ReadNetPara -> ReadSrvCfgPara -> ReadSrvProc`
    - 还解释了离线从 `$STARENV/dbmap/` 读取的分支
  - 同时携带了一个 `todo_write` 工具调用，用来把“总结 RTDB 进程数据来源”标记为完成

- `msg_e7378e2c-9288-40ad-94b0-1fffe3619930`
  - 是 `todo_write` 返回后的下一次 `model.end`
  - 内容只有：
    - `总结完毕！如果还有其他想深入了解的部分……`
  - 它是一次 `end_turn`
  - 因而成了当前 run 的“最后一条 AI 消息”

最终飞书侧回发的是这句短收尾，而不是前面那条完整回答。

### 案例二：starsrv 启动进程加载来源问题

在日志中同样能看到：

- 模型先输出了一大段正文，分析 `starsrv` / `csisvc` 如何读取和启动进程
- 这条正文后面又带了 `todo_write`
- `todo_write` 完成后，模型最后补了一句：
  - `你可以在数据库中尝试用以下方式查找这张表，避免编码问题导致的搜索失败。`

由于飞书桥接仍然只取最后一条 AI 消息，于是用户只看到了这句不完整、且容易串题的收尾。

## 相关代码位置

### 1. 飞书桥接层

文件：

- [backend/src/agent_lark.py](D:/ai/xu-agent/backend/src/agent_lark.py)

当前逻辑：

- `extract_final_ai_text(result)` 会倒序扫描 `result["messages"]`
- 找到第一条非空 `AIMessage` 就直接返回
- `_process_event_locked()` 再把这个返回值发回飞书

也就是说，这里实际上采用的是：

- “最后一条 AI 消息优先”

而不是：

- “最后一条有实质内容的用户可见回复优先”

### 2. QQ 桥接层

文件：

- [backend/src/agent_botpy.py](D:/ai/xu-agent/backend/src/agent_botpy.py)

这里也有同类最终回复提取逻辑，因此同样存在被短收尾覆盖正文的风险。

## 为什么会触发

### 1. `todo_write` 出现在最终答复之后

如果模型先写出了完整答案，又顺手调用 `todo_write` 标记完成，那么框架必须继续执行工具。

工具执行完以后，模型会被再次调用一轮，而这轮往往只剩一句收尾。

### 2. 桥接层只看最后一条 AIMessage

只要最后一轮返回的是短句，它就会覆盖前面真正的正文。

## 影响

1. 用户看到的回复信息量严重缩水。
2. 容易误判为模型没答、答偏、或者日志/前端丢消息。
3. 同类问题在飞书与 QQ 等渠道中都可能复现。

## 已确认的结论

这不是：

- 模型没有生成正文
- LangGraph 丢失了消息
- 日志写漏了内容

而是：

- **渠道桥接层错误地把最后一条 AI 收尾消息当成了最终回复**

## 建议修复方向

### 方向一：修复最终回复提取逻辑

不要简单取“最后一条 AIMessage”，而要优先取：

1. 最后一条 **有实质内容** 的 AI 回复
2. 如果存在“正文 + tool_call”的 AI 消息，应优先保留正文
3. 对纯收尾套话做降权或忽略，例如：
   - `总结完毕`
   - `如果还有其他问题随时告诉我`
   - `你可以继续……`

### 方向二：约束 `todo_write` 的调用时机

已在 `backend/src/agent_prompt.py` 的 `PROMPT_SECTIONS["todo"]` 中补充约束：

1. 简单问答不要调用 `todo_write`
2. 如果要更新 todo，必须先更新，再写最终回复
3. 一旦开始撰写最终答案，就不要再调用 `todo_write`
4. 不要在完整答案后追加 `todo_write` 只是为了标记完成或补一句总结

### 方向三：桥接层与 Agent 层双保险

更稳妥的做法是两层一起做：

1. 提示词减少“正文后再调工具”的概率
2. 桥接层保证即便模型仍然这么做，也不会把最终正文覆盖掉

## 排查同类问题时的检查点

如果后续再次遇到“飞书只回了半句”的问题，可按下面顺序快速检查：

1. 看 `backend/logs/agent.jsonl` 中是否先有一条长正文的 `model.end`
2. 看这条长正文是否同时带了 `todo_write` 或其他工具调用
3. 看工具返回后是否又出现一条很短的 `end_turn`
4. 看桥接层是否仍然使用“倒序取最后一条 AIMessage”的逻辑

只要这四步都成立，基本就可以确认为同类问题。
